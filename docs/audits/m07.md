## Phase 1 — Bugs

### B1 (High) — Concurrent single/bulk create returns 409 instead of being idempotent
create and the per-row loop in createBulk use a check-then-act pattern (findExistingAttendance
then save). Under two concurrent requests for the same (session, student) both reads return
null, both INSERT, and the second hits the partial unique index
uq_attendances_session_student_active, raising Postgres 23505, which QueryFailedErrorFilter maps
to 409 Duplicate attendance. This violates the milestone idempotency contract (plan 7.3/7.4).
The DB index prevents duplicate rows, but the service does not translate the race into the
return-existing-row outcome; it surfaces 409. Real FE path in M15 (double-tap, offline replay,
two devices). Area #1 the user flagged: unique constraint present but NO ON CONFLICT/upsert.
Fix: INSERT ... ON CONFLICT (session_id, student_id) WHERE deleted_at IS NULL DO NOTHING
RETURNING *, then re-select, so the conflict path returns the existing row.

### B2 (High) — Single create does not validate the student belongs to the teacher
create validates session ownership but studentId is taken from the body and only checked
against the users FK (any user) and against enrollment (snapshot only). A teacher can create an
attendance row referencing a foreign teacher student or any user UUID, a cross-tenant PII
linkage. is_enrolled_class is false (guest) so it looks like a walk-in, but the user is real and
not owned. e2e Case 6 exercises an unenrolled student without asserting ownership. Scope the
student to instructor_id = teacherId before insert. See S1; release blocker per CLAUDE.md.

### B3 (Low) — Session soft-delete not filtered on any path
findOne/findAll/resolveSession filter c.deleted_at IS NULL but never s.deleted_at IS NULL.
Attendance can be created against a soft-deleted session, and rows for one remain mutable.
Add s.deleted_at IS NULL if soft-deleted sessions should be read-only.

### B4 (Low) — mark* shortcuts have no status-transition or session-lifecycle guard
mark methods overwrite status unconditionally; no guard against marking a row whose session has
not started (M6 has start/end lifecycle, attendance ignores it). Likely OK for M7; flag it.

### B5 (Info) — Test gap: no concurrency test for idempotency
e2e covers sequential double-create only; no Promise.all concurrent case. Given B1 it would
currently fail (409). Add once B1 is fixed. The user explicitly asked about concurrent bulk.

### B6 (Info) — findAll/findOne have no pagination
findAll returns unbounded getMany() when no filter is supplied. Note for M9 (plan 9.4).

## Phase 2 — Security

### S1 (High) — Teacher-isolation correct on read/mutate, cross-tenant linkage via create (B2)
Isolation through session-class-teacher is correct and consistent on every read/mutate endpoint:
resolveSession, findOne, findAll all innerJoin s.class c and filter c.teacher_id = :teacherId,
returning 404 (never 403). Verified for GET /:id, GET /, all four mark*, updateNotes, POST /,
POST /bulk. e2e Case 5 covers the 404 paths. The only hole is B2 (studentId not teacher-scoped
on single-create), crossing the tenant boundary on PII. Treat B2/S1 as a blocker.

### S2 (Pass) — is_enrolled_class audit-snapshot invariant is a code-level guarantee
Area #2 verified at code level, not just test-covered. Set at insert in create (via
isStudentEnrolledInClass) and in createBulk (hardcoded true, loop iterates only active
enrollments). No read path recomputes or writes it: findOne/findAll only select; mark* set only
status+checkedInAt; updateNotes sets only notes. Never reassigned after insert anywhere.
Idempotent-return branches return the stored entity untouched. Enrollment lookups rely on
TypeORM default soft-delete exclusion (ClassEnrollment has DeleteDateColumn), so the snapshot
reflects active enrollment at insert time. Correct.

### S3 (Pass) — PII / response shape
User has Exclude on password and deletedAt; Attendance has Exclude on deletedAt. Global
ClassSerializerInterceptor means neither leaks; e2e confirms. Queries use innerJoin (NOT
innerJoinAndSelect), so student/session relations are not hydrated; only studentId/sessionId
are returned. No student PII (name, belt, birthday, registry) is embedded in responses.

### S4 (Pass) — No try/catch on PG error codes
Zero try/catch around Postgres codes. The three new constraints (409 dup, 422 fk session,
422 fk student) are registered in the QueryFailedErrorFilter map. Compliant. (B1 is a
correctness issue with relying on this for idempotency, not a try/catch violation.)

### S5 (Low) — Logging / PII redaction not verified for new surface
No new logging in the module; nothing logs studentId/notes. Redaction config unchanged. Notes
are free-text PII and are never logged; no tokens logged. Note: redaction list does not cover
notes, so a future error path logging the request body could expose it. Informational.

### S6 (Info) — DTO validation completeness
CreateAttendanceDto: sessionId/studentId IsUUID v4, notes optional MaxLength 500.
BulkCreateAttendanceDto: sessionId IsUUID v4. QueryAttendancesDto: optional UUIDs + IsEnum;
invalid status gives 422 (e2e Case 4). UpdateAttendanceNotesDto: notes optional string-or-null,
IsOptional+IsString+MaxLength 500; explicit null clears notes (controller maps notes ?? null).
No bug.

### S7 (Info) — Migration up/down reversibility
up creates table + partial unique index + 3 plain indexes, named constraints; down drops
indexes then table with IF EXISTS. Reversible and idempotent. FK ON DELETE CASCADE on session
and student is intentional. Cascading delete from users would hard-delete audit rows, but the
soft-delete-only policy makes a hard user delete unreachable in practice. Informational.

## Suggested suppressions
- S4 try-catch invariant: confirmed compliant; nothing to suppress.
- B4 status-transition guard: if unconditional mark* is intended M7 behavior (status cycling is
  a deliberate FE affordance in M15), record in accepted-findings.md so it is not re-raised.
- S7 FK CASCADE on student: accepted-by-design (soft-delete-only makes hard delete unreachable).

## Blockers
- B1 (High): Concurrent create is not idempotent; returns 409 on race instead of the existing
  row. Fix with ON CONFLICT DO NOTHING upsert + re-select. Idempotency under concurrency is the
  milestone named guarantee and a FE-critical path. Blocks merge.
- B2 / S1 (High): Single-create does not teacher-scope studentId, allowing a teacher session to
  bind to a foreign teacher student. Teacher-isolation is a release blocker per CLAUDE.md. Scope
  studentId to instructor_id = teacherId before insert. Blocks merge.

Non-blocking but recommended before M9: B4 (status-transition guard — accepted as M7 behavior),
B6 (pagination).

## Resolution (post-audit fixes, feat/m07-attendances)

### B1 (High) — RESOLVED
create and the createBulk per-row loop now insert via the query builder with .orIgnore()
(ON CONFLICT DO NOTHING against uq_attendances_session_student_active), then re-select through a
new getExistingOrThrow() helper. A racing duplicate is absorbed and the existing row is returned
with 201 — no 23505, no 409. The is_enrolled_class snapshot is unaffected: it is still computed
once before the insert, and the re-select returns the stored value (S2 invariant intact).

### B2 / S1 (High) — RESOLVED
create now calls a new resolveStudent(studentId, teacherId) helper that loads the student scoped
to instructor_id = teacherId and throws 404 on miss (mirrors ClassesService.enroll). User repo
registered in AttendancesModule's TypeOrmModule.forFeature. createBulk was already safe (students
derived from the teacher-owned class's enrollments).

### B3 (Low) — RESOLVED
Added s.deleted_at IS NULL to resolveSession, findOne, and findAll, so attendance cannot be
created against or read/mutated on a soft-deleted session.

### B5 (Info) — RESOLVED
Added an e2e concurrency test (Case 6): 4 parallel POST /attendances all return 201 with one
shared id and exactly one DB row. Also added a Case 5 test asserting a teacher attaching another
teacher's student returns 404. Full attendances e2e suite: 27/27 green. typecheck clean.

Remaining open: B4, B6 (non-blocking, deferred to M9 per above).
