**Date**: 2026-05-20
**Auditor**: milestone-auditor + security-review

## Phase 1 — Bugs

### B1 — TOCTOU race in `update` between tenant guard and mutation — LOW

`StudentsService.update` calls `findOne(id, currentTeacherId)` (a SELECT), then `usersService.update(id, dto)` (an UPDATE) as two separate round trips with no transaction or lock. Between the two calls another request can soft-delete the student. The `usersRepository.update` call then silently succeeds on a deleted row, returning the entity with `deletedAt` populated rather than throwing. The caller gets a 200 with a soft-deleted entity.

The same race exists in `softDelete`, where two simultaneous DELETE calls both pass the tenant guard and call `usersService.softDelete(id)` — harmless on idempotent soft-delete, but the second call returns 204 even after the row is already deleted.

**Fix shape (deferred to M9 or a follow-up)**: wrap the guard + mutation in a TypeORM transaction, or perform the ownership check inside the UPDATE WHERE clause (single round-trip, atomic by construction).

### B2 — `student!` non-null assertion in `findOne` — MINOR

`students.service.ts:96` uses `return student!` after the `belongsToTeacher` type guard throws. The assertion is logically correct, but the `!` operator is a project invariant violation (no type laundering). Because `belongsToTeacher` is declared as `student is User`, TypeScript should narrow `student` to `User` without the assertion. The assertion is redundant and should be removed.

### B3 — `validateExclusionFilters` is not tenant-scoped — NOTED (acknowledged)

`students.service.ts:180-199` probes for the existence of a class/session row without checking that the class/session belongs to the calling teacher. A teacher can therefore use a UUID from another teacher's class as a `notEnrolledInClass` filter and receive a 200 (rather than 404) if the UUID resolves to a real row in the `classes` table. This leaks existence of cross-tenant class rows.

Explicitly acknowledged in the code comment: "Tenant scoping on these lookups lands in M5/M6 when those modules ship." Deferred and documented. Not a blocker for M4; must be addressed before M9.

### B4 — `QueryUsersDto.name` and `registry` lack `@MaxLength` — LOW

`apps/api/src/users/dto/query-users.dto.ts` declares `name` and `registry` as `@IsString()` with no length ceiling. `QueryStudentsDto` adds `@MaxLength(120)` and `@MaxLength(40)` respectively. Any future callers of `findByRole` passing `QueryUsersDto` directly would accept unbounded strings. The LIKE pattern itself is parameterized (no SQL injection), but very long patterns can cause CPU amplification on the `ILIKE` expression. Carry-forward to be fixed before M9.

### B5 — Unit test does not assert positive path for exclusion filters — LOW

`students.service.spec.ts` tests that a non-existent class UUID throws `NotFoundException`. It does not explicitly assert that when the probe returns a result, `findByRole` is still called and no exception is thrown. Minor test-gap; low severity.

### B6 — E2e test accepts 500 as valid for `notEnrolledInClass` — INFO

`students.e2e-spec.ts:442`: `expect([404, 500]).toContain(res.status)`. Accepting 500 in an e2e assertion is a test-hygiene issue. The comment explains this is a M4 limitation (classes table not yet created). Must be tightened in M5.

## Phase 2 — Security

### S1 — Teacher-isolation boundary: PASS

All five operations are correctly teacher-scoped:
- **create**: instructor forced from `currentTeacherId`, never from DTO. `ValidationPipe whitelist + forbidNonWhitelisted` rejects any attempt to pass `instructor` in the body (e2e confirmed).
- **findAll**: passes `currentTeacherId` to `usersService.findByRole`, which ANDs `user.instructor_id = :teacherId`. Filter applied before `getCount()` and `getMany()`.
- **findOne**: `usersRepository.findOne` fetches without teacher scope, then `belongsToTeacher` checks ownership. Cross-teacher access throws `NotFoundException` (not `ForbiddenException`). Contract: 404 never 403. PASS.
- **update / softDelete / restore**: all gate through `findOne` or equivalent inline check before mutating. PASS.
- E2e covers GET, PATCH, DELETE, restore isolation. All assert 404 not 403. PASS.

**Teacher-isolation invariant: PASS — not a blocker.**

### S2 — Belt-rank CASE expression: no SQL injection vector — PASS

`users.service.ts:196-208` interpolates `Belt.*` enum members resolved at compile time. The `Belt` enum is a string enum defined in `packages/contracts`; its values are fixed TypeScript constants, not user-supplied strings. No path exists for user input to reach this template. PASS.

### S3 — `sortBy` interpolation in `applySorting` — LOW (not injection, fragile)

`users.service.ts:212`: `qb.orderBy(\`user.${sortBy}\`, sortOrder)`. Both `sortBy` and `sortOrder` are validated by `@IsIn(...)` in the DTO before reaching the service. Risk is zero as long as `sortBy` stays whitelist-validated. The concern is that if a future refactor loosens the whitelist or the parameter type is widened to `string`, this becomes a column-injection vector.

**Recommendation**: add a compile-time exhaustiveness check or use a mapping object instead of direct interpolation to make the constraint explicit.

### S4 — `validateExclusionFilters` raw SQL: parameterized, no injection — PASS

```ts
`SELECT 1 FROM classes WHERE id = $1 LIMIT 1`,
[query.notEnrolledInClass],
```
Both queries use positional parameters. The values are UUID-validated by `@IsUUID('4')` in the DTO. PASS.

### S5 — No `try/catch` on Postgres error codes in the students module — PASS

Neither `StudentsService` nor any M4 code contains `try/catch` blocks that inspect Postgres error codes. Constraint violations are correctly left to propagate to `QueryFailedErrorFilter`. PASS.

### S6 — Multi-tenancy completeness — PASS

Every query in M4 is scoped by `currentUser.id`. The `validateExclusionFilters` helper is the only query without tenant scoping, and that gap is explicitly acknowledged and deferred (see B3). PASS for M4 scope.

### S7 — No JWT secret fallback — PASS

No new JWT strategy code in M4. The M3b audit confirmed `configService.getOrThrow` with no fallback. PASS.

### S8 — Optional `password` field on `CreateStudentDto` — INFO

`create-student.dto.ts` exposes an optional `password` field marked for future use. Current `AuthService.validateCredentials` checks for the `TEACHER` role before issuing tokens, so students cannot log in. Latent risk if the login gate is ever relaxed. Must re-audit when student portal scope is revisited (M12+).

### S9 — Password exclusion from responses — PASS

`@Exclude()` on the `User` entity field correctly prevents password leakage via `ClassSerializerInterceptor`. E2e confirms this. PASS.

### S10 — Dependency CVEs — LOW

`pnpm audit` reports 8 vulnerabilities:
- 6× `HIGH`: `tar < 7.5.8` via `bcrypt@5.1.1 → @mapbox/node-pre-gyp@1.0.11 → tar@6.2.1`. Path-traversal CVEs in `tar`. Used only during `bcrypt`'s native module build at install time, not at runtime. No production-runtime-reachable risk.
- 1× `MODERATE`: `esbuild` via `vitest` in `packages/contracts` — dev/test only.
- 1× `MODERATE`: `vite < 6.4.2` via `vitest@2.1.9` in `packages/contracts` — dev/test only.

No production-runtime-reachable CVEs. LOW.

UUIDs are unguessable per project precedents; the acknowledged deferred finding (B3) covers the full tenant-scoping gap when M5/M6 land. **Confidence: 5/10 — not filed.**

All Phase 2 findings from the first pass are confirmed. **No additional security blockers.**
