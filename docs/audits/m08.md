## Phase 1 - Bugs

### B1 (Medium) - RUN_MIGRATIONS Joi default is true, enabling silent auto-migration in production

File: apps/api/src/config/configuration.ts, line 8

The Joi schema gives RUN_MIGRATIONS a default of true. The main.ts migration gate therefore runs migrations on every startup in any environment where RUN_MIGRATIONS is not explicitly set to false. A production container deployed without this variable set will silently apply all pending migrations on boot, potentially applying destructive schema changes before the operator is aware. The code comment (gated by RUN_MIGRATIONS=true) implies the default should be false - opt-in, not opt-out.

Recommendation: Change Joi default to false. The dev .env already sets RUN_MIGRATIONS=true explicitly, so local dev is unaffected.

---

### B2 (Low) - configuration.ts reads boolean env vars via string comparison, not Joi-coerced values

File: apps/api/src/config/configuration.ts, lines 40, 50, 61

The factory re-reads process.env directly with === true string comparisons after Joi has validated and coerced the values. A non-standard truthy string (e.g. RUN_MIGRATIONS=1) passes Joi validation but evaluates to false in the factory, silently leaving the flag unset. Affects RUN_MIGRATIONS, SWAGGER_ENABLED, and DB_LOGGING. No current breakage since all .env files use the string true, but breaks the single-source-of-truth principle.

Recommendation: Document that only the string true is accepted. Existing behavior is stable.

---

### B3 (Low) - SEED_TEACHER_PASSWORD exported from seed module, embeds plaintext credential in compiled dist

File: apps/api/db/seeds/run-seed.ts, line 50

The constant uses the export keyword. The plaintext teacher password therefore appears in dist/db/seeds/run-seed.js and in the generated dist/db/seeds/run-seed.d.ts (already visible in the diff). The production guard prevents execution, but the compiled dist that ships in a production image contains the credential string.

Recommendation: Remove the export keyword. E2e tests needing this credential should read it from .env.test or a test-support fixture, not the seed module.

---

### B4 (Low) - Health e2e test covers only the happy path; 503 failure path untested

File: apps/api/test/health.e2e-spec.ts

Tests assert GET /health returns 200 with status: ok and database indicator as up. No test for the 503 failure path that Terminus returns when the DB is unreachable. This is the shape FE health-polling logic will depend on in M16b.

Recommendation: Add a test that mocks TypeOrmHealthIndicator.pingCheck to throw and asserts 503 with res.body.status === error.

## Phase 2 - Security

### S1 (High) - SWAGGER_ENABLED Joi default is true; production deploy without the flag exposes full API surface

File: apps/api/src/config/configuration.ts, line 24

The Joi schema defaults SWAGGER_ENABLED to true. In production, if the operator does not explicitly set SWAGGER_ENABLED=false, the Swagger UI is mounted at /api and the JSON schema is served at /api-json. This exposes every endpoint parameter shape, auth requirement, response schema, and tag grouping to unauthenticated users - a reconnaissance aid for attackers.

The main.ts guard (if config.get(features.swaggerEnabled) === true) is structurally correct but undermined by default(true) in the Joi schema.

Recommendation: Change Joi default to false. Swagger should be opt-in (SWAGGER_ENABLED=true) for dev environments, not opt-out for production. The dev .env already has SWAGGER_ENABLED=true explicitly.

---

### S2 (Medium) - Health endpoint rate-limited at 100/60s; load-balancer probes will get 429 in clustered production

File: apps/api/src/health/health.controller.ts; apps/api/src/app.module.ts lines 117-119

The ThrottlerGuard from @nestjs/throttler does not read IS_PUBLIC_KEY metadata. Its shouldSkip() base implementation returns false unconditionally (confirmed by reading node_modules/@nestjs/throttler/dist/throttler.guard.js line 99). Therefore @Public() routes ARE throttled by the global 100/60s default. This is the correct security posture - public routes should be throttled.

However, GET /health is a load-balancer probe endpoint. In production with autoscaling, an orchestrator (AWS ALB, k8s liveness probe) probes every instance at a standard interval (often 10s). Multiple probes from multiple sources can exceed 100 requests per 60 seconds. When the limit is hit, the orchestrator receives 429 and may mark the instance as unhealthy, removing it from rotation despite the app being healthy.

The plan states @Public() routes should skip auth but still be rate-limited - this is satisfied. But the health endpoint specifically needs either a higher per-route limit or an explicit @SkipThrottle().

Recommendation: Add @SkipThrottle() to HealthController.check() handler, or configure a significantly higher per-route throttle limit. Update plan comment to clarify that @Public() bypasses only JwtAuthGuard, not ThrottlerGuard.

---

### S3 (Low) - Migration runner has no explicit NODE_ENV !== test guard despite code comment claiming it

File: apps/api/src/main.ts, lines 28-31

Comment states NEVER runs in test (NODE_ENV=test) but the actual condition is only RUN_MIGRATIONS === true with no NODE_ENV check. If .env.test were ever modified to include RUN_MIGRATIONS=true, the migration runner would race with the e2e suites own ds.runMigrations() call in beforeAll, potentially applying migrations twice against postgres-test.

Recommendation: Add process.env.NODE_ENV !== test to the migration-runner condition, or update the comment to accurately say that .env.test does not set RUN_MIGRATIONS.

### S4 (Low) - Dependency CVEs in build-time and devDep chain

Source: pnpm audit - 8 vulnerabilities (2 moderate, 6 high)

Six high-severity CVEs in tar via apps/api -> bcrypt@5.1.1 -> @mapbox/node-pre-gyp@1.0.11 -> tar@6.2.1: GHSA-34x7-hfp2-rc4v, GHSA-8qq5-rm4j-mr97, GHSA-83g3-92jg-28cx, GHSA-qffp-2rhf-9h96, GHSA-9ppj-qmqm-q256, GHSA-67mh-4wv8-2f99.

Two moderate CVEs via packages/contracts -> vitest@2.1.9: esbuild GHSA-67mh-4wv8-2f99 (dev server request forwarding), vite GHSA-4w7w-66w2-5vf9 (path traversal in .map files).

Assessment: @mapbox/node-pre-gyp is a build-time native binary downloader, not present in the production runtime. The esbuild and vite CVEs are in test devDependencies with no production deployment path. No runtime attack surface in either chain.

Recommendation: Accept and record in accepted-findings.md. Track bcrypt for an upgrade that drops node-pre-gyp.

## Blockers

Both issues must be resolved before this PR merges.

### BLOCKER 1 - SWAGGER_ENABLED Joi default is true (S1 - High)

Any production deployment where SWAGGER_ENABLED is not explicitly set will publicly serve the Swagger UI and the full OpenAPI JSON spec. Direct information-disclosure risk with zero friction to exploit.

Fix (one line, apps/api/src/config/configuration.ts line 24):
  SWAGGER_ENABLED: Joi.boolean().default(false),

The dev .env already has SWAGGER_ENABLED=true explicitly, so local dev is unaffected.
Fix owner: api-developer.

---

### BLOCKER 2 - RUN_MIGRATIONS Joi default is true (B1 - Medium)

Any deployment without RUN_MIGRATIONS=false will auto-migrate on boot. For a production database this can apply destructive schema changes without operator awareness. The gate must be opt-in.

Fix (one line, apps/api/src/config/configuration.ts line 8):
  RUN_MIGRATIONS: Joi.boolean().default(false),

The dev .env already sets RUN_MIGRATIONS=true explicitly, so local dev is unaffected.
Fix owner: api-developer.
