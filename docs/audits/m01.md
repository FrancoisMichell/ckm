## Phase 1 — Bugs

### BL1 — Blocker: pnpm lint fails for @ckm/contracts (no ESLint config)
- File: packages/contracts/package.json:17 declares a "lint": "eslint src" script; no eslint.config.js / .mjs / .cjs exists in the package or at the repo root. @ckm/eslint-config/index.js is present but is never imported — ESLint v9 only auto-discovers a flat config from the package or an ancestor directory.
- Symptom (verbatim from pnpm lint):
  > ESLint couldn't find an eslint.config.(js|mjs|cjs) file.
  > From ESLint v9.0.0, the default configuration file is now eslint.config.js.
- Impact: CI step Lint (.github/workflows/ci.yml:42-43) will fail on every push to this branch — including the M1 PR.
- Two equally valid fixes (pick one in the next commit, not in this audit):
  1. Add packages/contracts/eslint.config.js that re-exports the shared @ckm/eslint-config (module.exports = require('@ckm/eslint-config');). Preferred — matches the intent of M0 eslint-config skeleton.
  2. Drop the lint script from packages/contracts/package.json until ESLint is properly wired in M2. (Regression — the M0 audit already noted CI test would silently no-op without scripts; doing the same for lint just defers the problem.)
- Severity: blocker (CI breakage on the PR).

### B1 — High: getSessionStatus field naming diverges from the HTTP transport
- File: packages/contracts/src/session.ts:13-24
- SessionForStatus uses snake_case (start_time, end_time, deleted_at) but every documented response shape on the HTTP API is camelCase: docs/api/03-api.md:152-154,248-250,269 uses startTime, endTime; docs/api/05-architecture-and-conventions.md:361 confirms deletedAt on serialized entities. The FE will receive { startTime, endTime, deletedAt } from openapi-fetch and would have to manually remap keys to call this helper — which is precisely the friction the shared helper exists to eliminate.
- The contracts package is the single source of truth for cross-stack helpers; either the helper takes the camelCase wire shape (recommended — it will match the M10 generated OpenAPI types), or it accepts both via a typed adapter. Snake_case here implies the helper is for raw DB rows, but services do not expose raw rows.
- Note: the existing __tests__/session.test.ts is internally consistent with the current snake_case shape, so the test green light is meaningless for the consumer mismatch.
- Severity: high (forward-incompatible with M10 generated types and the M14 sessions UI). Fix is one search-and-replace plus a test rename; cheaper now than after the FE imports it.

### B2 — Medium: contracts vs api enum name divergence (UserRoleName vs UserRoleType)
- File: packages/contracts/src/attendance.ts:9-12 exports enum UserRoleName { TEACHER, STUDENT }.
- docs/api/02-domain-model.md:18, docs/api/03-api.md:246, docs/api/04-auth-and-rbac.md:82,100, docs/api/07-testing.md:64, root CLAUDE.md:147, and .claude/agents/api-developer.md:52 all spell it UserRoleType.
- The plan + contracts-keeper agent + the package itself say UserRoleName. Both names refer to the same enum.
- This will land as either (a) a duplicated enum on the api side — two sources of truth, exactly what packages/contracts exists to prevent — or (b) the api file aliases (export { UserRoleName as UserRoleType }), which is fine but must be conscious.
- Recommend: keep UserRoleName (per plan), and in M3a have api-developer re-export the contracts enum as UserRoleType for the api naming convention, plus a doc note. Decision should be recorded before M3a starts.
- Severity: medium (will not bite until M3a, but unresolved it guarantees one or the other agent reinvents the enum).

### B3 — Medium: UserRoleName enum lives in src/attendance.ts
- File: packages/contracts/src/attendance.ts
- File is named for the AttendanceStatus enum but also exports UserRoleName. Discoverability is poor — neither the file name nor the barrel ordering signals that user roles are there. Future readers will look in a hypothetical user.ts or roles.ts first.
- No runtime impact. Move to src/user.ts (or rename the file to src/enums.ts and consolidate all four enums) before any external consumer cements the import path. Easy now; harder after M3a.
- Severity: medium (organizational; cheap to fix before merge).

### B4 — Medium: package exports map is CJS-only; the FE is ESM
- File: packages/contracts/package.json:7-12 — exports["."] declares only types and require; no import condition.
- The package emits CommonJS (tsconfig extends @ckm/tsconfig/node.json which sets module: Node16; no "type": "module" in package.json, so tsc defaults to CJS under Node16). Verified — dist/index.js opens with "use strict" and __createBinding. That works for NestJS (CJS). It will NOT cleanly work for Vite/apps/web in dev — Vite resolves the import condition of exports, finds none, and either errors or falls back inconsistently.
- Today this is latent (no consumer yet). Two equally valid resolutions:
  1. Dual emit (tsup or two tsc invocations) plus both import + require conditions in exports.
  2. Ship ESM-only ("type": "module" + module: NodeNext); both Nest 11 and Vite 7 can consume ESM. Forward-aligned with the rest of the stack.
- Either way, the FE-first consumer (M10) will surface this. Recommend deciding now rather than at M10 under deadline pressure.
- Severity: medium (forward-looking; not a build break today because no consumer imports @ckm/contracts yet).

### B5 — Low: BELT_ORDER.indexOf is O(n) per call
- File: packages/contracts/src/belt.ts:36-38 (working-tree version — replaces the previous O(1) BELT_RANK lookup map).
- The committed (HEAD) implementation used a Record<Belt, number> lookup (O(1)). The working-tree change (B6) reintroduces an O(n) scan on each call, and compareBelts calls it twice per comparison. For 7-element arrays this is fine; for an attendance list of 100+ students sorted by belt it is 200 indexOf scans against a 7-element array per sort — still cheap, but an algorithmic-complexity regression for no obvious gain.
- The legitimate win of the new code is a single source of truth (BELT_ORDER removes the WHITE=1 / BLACK=7 list duplication). The rank could be derived once on module init and still preserve O(1) — build a Map<Belt, number> from BELT_ORDER.map((b, i) => [b, i + 1]) and look up against that.
- Severity: low (perf is fine at expected scale; flagged because the refactor traded O(1) for O(n) without comment).
