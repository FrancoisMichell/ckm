## Phase 1 — Bugs

### 1. Checkpoint fixes — verified to hold

- **H1 (bcrypt timing oracle)** — `auth.service.ts:57-63` initialises `dummyPasswordHash` in `onModuleInit` using `bcrypt.hash('invalid', rounds)`. `validateCredentials` at lines 90-110 unconditionally calls `passwordService.compare(password, hashToCompare)` before any 401, where `hashToCompare = user && user.password ? user.password : this.dummyPasswordHash`. Unit tests at `auth.service.spec.ts:166-197` assert bcrypt runs on both the missing-user and null-password paths. **Fix is intact.**
- **H2 (atomic rotation)** — `auth.service.ts:246-263` wraps the new-row insert and the old-row update in `this.refreshTokenRepo.manager.transaction(async (em) => { ... })`. Spec at `auth.service.spec.ts:492-513` asserts `transactionMock` is invoked once and both writes flow through the `EntityManager`. **Fix is intact.**
- **M1 (pino redaction)** — `pino.config.ts:25-37` lists `*.access_token`, `*.refresh_token`, `*.password`, `*.tokenHash`, `*.lookupHash`, `*.token_hash`, `*.lookup_hash`, `req.headers.authorization`, `req.headers.cookie`. **Fix is intact.**
- **L1 (dead deletedAt branch)** — removed. `validateCredentials` no longer references `user.deletedAt`.
- **L2 (silent revocation failure)** — `auth.service.ts:210-222` wraps `revokeFamilyById` in try/catch with `errorReporter.captureException` and still throws 401. Asserted at `auth.service.spec.ts:545-571`. **Fix is intact.**
- **I1 (comment drift)** — `app.module.ts:39-44` now references the 5/60s `@Throttle` override on `/auth/login` rather than M8. Cleaned up.

No regressions detected against the checkpoint findings.

### 2. Swagger module promised in 3b.7 plan but never wired — LOW

`docs/plan.md:483-485` (sub-step 3b.7) says: "**`AuthController` `/auth/{login,refresh,logout,me}` + Throttler 5/60s on `/login` + Swagger module enabled + e2e `auth.e2e-spec.ts`**".

`@nestjs/swagger` is listed in `apps/api/package.json:32`, and `SWAGGER_ENABLED` plus `swaggerEnabled` are wired through Joi and configuration (`configuration.ts:24,55`). But neither `apps/api/src/main.ts` nor `apps/api/src/common/setup-app.ts` actually constructs a `DocumentBuilder` / calls `SwaggerModule.setup(...)`. Grep across `apps/api/src` only finds `@nestjs/swagger` used in `common/dto/include-deleted.dto.ts` for `@ApiPropertyOptional`. The flag is dead.

LOW because no security surface depends on Swagger and most consumers of /auth in M3b are e2e tests, not the dev-facing UI. The contract regen flow (`pnpm openapi:generate`) presumably depends on this — verify before M4 starts or you cannot regenerate types when Students lands. Consider deferring to a one-line follow-up commit on this branch, or explicitly bumping the plan to mark this as carried into M8.2.

### 3. Refresh-rotation commits even when the user has been soft-deleted — LOW

`auth.service.ts:246-282`: the rotation transaction commits the new refresh-token row and revokes the old row **before** `usersService.findById(existingRow.userId)` is called at line 266. If the user has been soft-deleted between login and the rotation attempt, `findById` returns `null` and the function throws `UnauthorizedException('User no longer exists.')`.

DB left in an inconsistent intermediate state: the old row is revoked (correct) and a new, unrevoked, never-issued refresh-token row exists for a soft-deleted user. It is unreachable by the FE (the FE got a 401 and threw the token away), and `ON DELETE CASCADE` on `fk_refresh_tokens_user_id` will reap these on hard delete. But it muddies the otherwise-clean invariant "every row in `refresh_tokens` corresponds to a token a client has held".

Fix shape (deferred): pull the `findById` call **before** the transaction and abort if null. Cost: one extra SELECT before the TX.

LOW — no security impact, defensive cleanup only.

### 4. Stale-snapshot roles in JWT across demotion — INFO

`auth.service.ts:266-281`: after rotation, `signAccessToken` uses fresh DB roles. Good. But there is no symmetric protection for a teacher who has been **demoted** to STUDENT between login and refresh: the JWT is reissued correctly, but the user retains a 15-minute access token signed at login time with the old `roles: [TEACHER]`. Since there is no JWT denylist, a demoted teacher keeps teacher privileges for up to 15 minutes after demotion.

Documented bounded-window cost of stateless JWTs. Acceptable for M3b. Surface in M9 or when role-mutation endpoints land (M4+); those may want to invoke `revokeFamilyByUserId` to force a refresh-cycle within the access-token TTL.

### 5. Throttler fires before LocalAuthGuard — NOTED, not a bug

`auth.controller.ts:54-58`: `@Public() → @UseGuards(LocalAuthGuard) → @Throttle(...) → @HttpCode(200) → @Post('login')`. Nest's APP_GUARD chain runs registered guards before route-level guards, so `ThrottlerGuard` (APP_GUARD) fires before `LocalAuthGuard`. The decision ("invalid logins also consume the 5/60s budget") is correct on the actual runtime path. The e2e at `auth.e2e-spec.ts:281-294` confirms this (5 × 401 then 1 × 429 on bad credentials).

Explicit anti-credential-stuffing posture. A user who fat-fingers 5× will be locked out for ~60s. Intended; document in the runbook entry for M3b.

### 6. `@Throttle` decorator precedence — verified

Confirmed via `@nestjs/throttler@5+`: when a route handler has `@Throttle({ default: { limit, ttl } })`, the global `ThrottlerGuard` resolves the override and ignores the global limit for that route. `/auth/login` bounded by 5/60s only; `/auth/refresh` and `/auth/logout` bounded by the 100/60s global cap. **No bug.**

### 7. Test gap — no e2e on `JwtAuthGuard` 401 path — LOW

E2E covers happy-path login, bad-creds, logout flow, replay, and 429. It does not cover:
- a missing `Authorization` header on `/auth/me` (must 401 via `JwtAuthGuard`)
- a malformed/expired JWT on `/auth/me`
- a JWT whose `sub` corresponds to a soft-deleted user (the JwtStrategy does not re-validate against the DB, so this currently still succeeds — relevant for M4+ when soft-delete on user lands)

Unit tests at `jwt-auth.guard.spec.ts` and `jwt.strategy.spec.ts` cover the bypass logic and `validate()` shape. Gap is at e2e integration only. Recommend adding two negative-path cases or carrying to M9's isolation suite.

### 8. `TROTTLE_TEST` typo + in-memory throttler cross-file caveat — INFO

`auth.e2e-spec.ts:278` uses `registry: 'TROTTLE_TEST'` (should be `THROTTLE_TEST`). Cosmetic — never reaches the DB. Also note: Nest's default `ThrottlerStorageService` is per-process. Each Jest test file gets a fresh `INestApplication`, so counter doesn't bleed across files. Safe today. If anyone moves to Redis-backed throttler in CI or shares an app across files, this test becomes flaky. Worth a comment in `app.e2e-helper.ts`.

## Phase 2 — Security

### 1. Auth boundary
- **JWT secret fallback**: absent. `app.module.ts:32`, `jwt.strategy.ts:25`, `auth.service.ts` all use `configService.getOrThrow<string>('jwt.secret')`. Joi at `configuration.ts:19` requires `min(32)`. No `process.env.JWT_SECRET ?? ...` anywhere.
- **`ignoreExpiration`**: `false` at `jwt.strategy.ts:24`.
- **`@Public()` on `/auth/login` and `/auth/refresh`**: correct; `/auth/logout` and `/auth/me` are JWT-protected.
- **`/auth/logout` JWT-protected** but does NOT verify the supplied `refresh_token` belongs to `req.user.id`. An attacker holding a victim's access token AND a sibling's refresh token could log the sibling out. Exploit requires two stolen secrets — worse compromise than the logout-DoS. NOTED, not a bug.

### 2. Refresh-token family revocation
`auth.service.ts:315-322` revokes by `family_id` via QueryBuilder UPDATE — every non-revoked row in the family is set to `revoked=true`. Replay path invokes inside try/catch + ErrorReporter (L2). E2E at `auth.e2e-spec.ts:215-262` verifies DB-level invariant. **PASS.**

### 3. ThrottlerModule `skipIf` escape hatch — tight enough

`app.module.ts:47-54`:
```ts
skipIf: (context) => {
  if (process.env['NODE_ENV'] !== 'test') return false;
  const req = context.switchToHttp().getRequest<Request>();
  return req.headers['x-test-skip-throttle'] === '1';
}
```

- **Fail-closed on env**: `NODE_ENV=production` or unset → returns `false`. Joi default of `'development'` validates but does not mutate `process.env`. **Cannot be enabled in production by header alone.**
- **No request-body driven bypass**: HTTP header only.
- **Header is well-named**: `x-test-skip-throttle` would be visible in log review.
- **Inert in dev**: dev with curl would have to set `NODE_ENV=test` AND send the header.

One concern (LOW): a deployment that ships with `NODE_ENV=test` (misconfigured Docker, wrong .env mounted) plus an attacker guessing the header bypasses rate limit. Defence in depth: also bind bypass to a per-deploy secret (`THROTTLE_TEST_TOKEN=<random>`, header must equal that secret). Cheap. **Suggested non-blocking hardening.**

Verdict: **adequate for current threat model.**

### 4. Duplicate `ErrorReporter` provider in `AuthModule` — LOW

`auth.module.ts:50` declares `{ provide: 'ErrorReporter', useClass: NoopErrorReporter }` as a module-local fallback. Required because `AppModule` does not mark its `'ErrorReporter'` provider as `@Global`; without the local binding, `AuthService`'s `@Inject('ErrorReporter')` cannot resolve.

Hazard: when a real reporter (e.g. Sentry) lands, **both bindings must change in lockstep.** A future agent editing `app.module.ts` alone will silently leave AuthModule using NoopErrorReporter.

**Recommended hardening**: promote the root provider into a `@Global() ErrorReporterModule` exported from `apps/api/src/common/error-reporter/error-reporter.module.ts`. AuthModule then drops the local binding. Single source of truth. Suitable follow-up on this branch or carried to M8.2.

### 5. Input validation
- `/auth/login`: `LoginDto` validates `registry: string ≥1`, `password: string ≥1`. ValidationPipe runs pre-`LocalAuthGuard`. Verified at `auth.controller.ts:65`.
- `/auth/refresh`, `/auth/logout`: `RefreshTokenDto.refresh_token: string ≥1`. **INFO**: no max length. bcrypt safely truncates to 72 bytes; SHA-256 lookup does not. Body parser default ~100 KB on Express. Consider `@MaxLength(200)` for tidy 422s and defence in depth.
- ValidationPipe configured `whitelist: true, forbidNonWhitelisted: true, transform: true, errorHttpStatusCode: 422` at `setup-app.ts:45-55`. Solid.

### 6. SQL injection vectors
None in M3b. The single raw SQL in the suite is `auth.e2e-spec.ts:255-257` using a parameterised query. `revokeFamilyById` QueryBuilder uses bound parameters. `applySorting` CASE in `users.service.ts:196-208` interpolates `Belt.*` enum values (compile-time constants). **PASS.**

### 7. Secret leakage in logs
Pino redaction paths cover all sensitive fields including snake_case mirrors and request headers. E2E asserts none appear in response bodies. **PASS.**

### 8. `password` / `deletedAt` exposure
`@Exclude()` on `user.password`, `user.deletedAt`, `refresh_token.tokenHash`, `refresh_token.lookupHash`. `ClassSerializerInterceptor` global. E2E asserts non-leakage. **PASS.**

### 9. Bearer token storage (FE)
Out of M3b scope. Backend issues tokens in response body only; no `Set-Cookie`. CORS `credentials: false`. **PASS for backend half.**

### 10. CORS
`setup-app.ts:35`: `app.enableCors({ origin: allowedOrigin, credentials: false })`. `ALLOWED_ORIGIN` required by Joi. **PASS.**

### 11. Throttler-on-failed-login amplification
In-memory, keyed by IP via default `getTracker`. NAT-pool attacker can sustain 5×N attempts. Acceptable for v1. For M9: add IP-based reverse-proxy cap (Caddy/Nginx) as a second layer.
