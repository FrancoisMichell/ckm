# M3a Audit — Users Data Layer

## Summary

The M3a data layer is largely well-constructed. The migration is idempotent and every constraint is named. The entity respects soft-delete and `@Exclude()` rules. The service avoids `try/catch` on Postgres error codes and has no `synchronize: true`. Two blockers must be resolved before merge.

---

## Bugs

**BUG-1 (BLOCKER): `UsersService.update` passes raw `Partial<User>` — plaintext password written to DB**

`users.service.ts` lines 69–71: `update()` calls `usersRepository.update(id, patch)` without checking or hashing `patch.password`. The `create()` method hashes correctly. Any future controller calling `usersService.update(id, { password: 'newPlain' })` will store the password in plaintext. Fix: hash `patch.password` before the `update` call (or add a dedicated `changePassword` method).

**BUG-2 (BLOCKER): `uq_users_registry` is a plain UNIQUE — blocks registry reuse after soft-delete**

The migration defines `CONSTRAINT "uq_users_registry" UNIQUE ("registry")` without a `WHERE deleted_at IS NULL` predicate. A soft-deleted user with `registry = 'T001'` blocks creation of a new user with the same registry permanently. The correct fix is a partial unique index:

```sql
CREATE UNIQUE INDEX "uq_users_registry_active" ON "users" ("registry")
  WHERE deleted_at IS NULL;
```

**BUG-3 (MODERATE): Exclusion filters in `applyExclusionFilters` lack `deleted_at IS NULL` on joined tables**

The LEFT JOINs to `class_enrollments` and `attendances` use raw table names. When those tables are created in M5/M7 as soft-deleted entities, the exclusion filters may incorrectly include users whose attendance/enrollment row is soft-deleted. Fix at arrival time in M5/M7.

**BUG-4 (MODERATE): `softDelete` / `restore` swallow `affected = 0` — silent no-op on missing ID**

Both methods discard the `UpdateResult`. A future controller calling `softDelete(nonExistentId)` will return 200 OK silently. Fix: inspect `affected` and throw `NotFoundException` when `affected === 0`.

**BUG-5 (MINOR): `update` does not detect "not found" from `affected`**

`usersRepository.update` result is discarded; `findById` returning `null` is the only signal. Correct in production behavior, but the unit test passes for the wrong reason.

**BUG-6 (MINOR): Belt sort secondary `addOrderBy('user.name', 'ASC')` ignores `sortOrder`**

The secondary sort is always `ASC` regardless of the requested `sortOrder`. May be intentional as a stable tie-breaker but is undocumented.

**BUG-7 (MINOR): `QueryUsersDto.belts` has no `@Transform` to split comma-separated input**

`?belts=white,blue` produces a 422 instead of splitting into an array. Only matters for non-standard clients.

## Security

**SEC-1 (BLOCKER): Same as BUG-1** — plaintext password storage via `update()`.

**SEC-2 (PASS): No JWT secret fallback** — `JWT_SECRET` required via Joi `.min(32).required()`, `getOrThrow` used throughout.

**SEC-3 (PASS): `@Exclude()` on `password` and `deletedAt`** — `ClassSerializerInterceptor` registered globally in `setupApp`.

**SEC-4 (PASS): No `try/catch` on Postgres error codes** — `QueryFailedErrorFilter` maps by constraint name.

**SEC-5 (PASS): Pino redaction** — `authorization`, `cookie`, `password`, `refresh_token` all redacted.

**SEC-6 (PASS): `synchronize: false`** — both `app.module.ts` and `datasource.ts`.

**SEC-7 (PASS): CORS with explicit `allowedOrigin`** — not a wildcard; `credentials: false`.

**SEC-8 (PASS): Helmet applied** — `app.use(helmet())` present in `setupApp`.

**SEC-9 (NOTE): Global filters instantiated with `new NoopErrorReporter()` rather than via DI** — swapping to a real Sentry reporter in `AppModule` won't affect the filters until `setupApp` is also updated.

**SEC-10 (NOTE): `.env.test` committed with a known test JWT secret** — intentional for deterministic e2e; secret must never be reused in production.

**SEC-11 (NOTE): `x-request-id` passthrough — no UUID sanitization** — a client can inject arbitrary strings into logs. Low severity for an internal system.

## Soft-delete Correctness

- `@DeleteDateColumn deletedAt` present; TypeORM automatically appends `deleted_at IS NULL` to QueryBuilder and `findOne` calls. No `withDeleted: true` appears in the service — correct.
- `softDelete` / `restore` use TypeORM native methods — correct.
- `IncludeDeletedDto` exists but is not wired into any endpoint in M3a — correct.
- **Issue (BUG-2):** `uq_users_registry` plain UNIQUE blocks registry reuse after soft-delete.
- **Forward issue (BUG-3):** Exclusion LEFT JOINs will need `deleted_at IS NULL` when M5/M7 tables land.

## Test Coverage

**Covered:**
- Happy-path CRUD — unit + e2e.
- Duplicate-registry → 409 — e2e.
- Soft-delete hides from `findById` — unit + e2e.
- Restore makes visible — e2e.
- Belt filter and belt-rank CASE sort — e2e.
- QueryBuilder filters (name, belt, notEnrolledInClass, notInSession, teacherId) — unit.
- `QueryFailedErrorFilter` constraint map for `uq_users_registry` — e2e.

**Gaps:**
- No test for `update` with a password field (BUG-1 vector).
- No test for soft-delete + re-create with same registry (BUG-2 vector).
- No teacher-isolation e2e: two teachers, each scoped, cannot see each other's students.
- No test that `softDelete` / `restore` on a non-existent ID errors (BUG-4 vector).
- No test for `createdAt` sort path.
- No explicit assertion that `findByRole` excludes soft-deleted users.

## Minor Notes

- `datasource.ts` uses `!` non-null assertions on env vars — fine for CLI usage but gives cryptic Postgres errors if vars are missing (bypasses Joi).
- `PasswordService` is registered as a provider in both `UsersModule` and `AppModule` — two singleton instances of a stateless utility. Clean up when `CommonModule` is extracted.
- e2e `afterEach` uses `DELETE FROM "users"` without WHERE — correct for test isolation via FK cascade.
