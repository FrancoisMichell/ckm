## Phase 1 - Bugs

### 1.1 WARN - startTime validation accepts semantically invalid times

**File**: apps/api/src/classes/dto/create-class.dto.ts line 52

The @Matches regex /^\d{2}:\d{2}$/ passes for values like 99:99 or 25:60. Format only is validated, not semantic range. The database column is VARCHAR(5) with no CHECK constraint, so invalid times persist without error.

**Impact**: ClassSession logic in M6 using start_time for scheduling could receive logically invalid values, causing silent arithmetic errors.

**Suggested fix**: tighten to a range-validating regex such as /^([01]\d|2[0-3]):[0-5]\d$/.

---

### 1.2 WARN - days array allows empty input

**File**: apps/api/src/classes/dto/create-class.dto.ts line 42

@IsArray() and @IsEnum(DayOfWeekValue, {each:true}) both pass for days: []. No @ArrayMinSize(1) present. A class with zero schedule days will silently persist to the database.

**Suggested fix**: add @ArrayMinSize(1) to CreateClassDto.days.

---

### 1.3 WARN - Dead code: day_of_week_enum Postgres type created but never used

**File**: apps/api/db/migrations/5-CreateClassesAndEnrollments.ts lines 22-32

The migration creates the day_of_week_enum Postgres type with seven values, but the classes.days column is TEXT[] not day_of_week_enum[]. The type is unreferenced by any column. down() drops it so rollback is safe. The unused type is a schema artifact that could confuse future migration authors and conflict if a later migration re-creates it.

**Suggested fix**: remove the CREATE TYPE / DROP TYPE pair, or change the column to day_of_week_enum[].

---

### 1.4 WARN - validateExclusionFilters checks bare existence, not teacher-scoped existence

**File**: apps/api/src/students/students.service.ts lines 179-195

notEnrolledInClass and notInSession are validated via parameterized SQL (SELECT 1 FROM classes WHERE id = placeholder) without teacher_id filter. Consequences:

- Teacher B can supply ?notEnrolledInClass=<uuid-of-teacher-A-class> and receive 200 (filter is a no-op). No 404 is raised for the cross-tenant probe.
- The query confirms to teacher B that the class UUID exists, leaking cross-tenant class ID existence.

Code comment acknowledges deferral: Tenant scoping on these lookups lands in M5/M6. Now that M5 is complete, this can be closed.

**Impact**: low-severity tenant isolation leak. Confirms cross-tenant class ID existence only; does not expose class data.

---

### 1.5 WARN - enroll() ownership checks execute outside the transaction (TOCTOU)

**File**: apps/api/src/classes/classes.service.ts lines 151-161

findOne(classId, currentTeacherId) and the student ownership check both execute before dataSource.transaction() starts. A concurrent soft-delete of the class between the check and the transaction body could allow enrollment into a deleted class. The transaction body does not re-verify class existence.

**Impact**: extremely low probability in a single-teacher deployment. The partial unique index guards enrollment deduplication independently.

**Suggested fix**: move ownership validation inside the transaction, or use SELECT ... FOR UPDATE on the class row.

---

### 1.6 INFO - Test gap: unenroll and restore cross-teacher isolation not e2e covered

**File**: apps/api/test/classes.e2e-spec.ts

The Teacher-isolation describe block covers GET (list), GET (one), PATCH, DELETE (class), GET enrollments, and cross-teacher enrollment attempt. Missing cases:
- Teacher B calling DELETE /classes/:classIdA/enrollments/:studentId where :classIdA belongs to teacher A.
- Teacher B calling POST /classes/:classIdA/restore where :classIdA belongs to teacher A.

Both paths call findOne(id, currentTeacherId) internally so the logic is correct. Gap is test coverage only.

### 1.7 WARN - Minor type laundering

**File**: apps/api/src/students/students.service.ts lines 48, 183, 195

Three casts: (1) instructor: { id: currentTeacherId } as User -- standard TypeORM partial-entity relation binding, no safer alternative. (2+3) as unknown[] on raw .query() returns -- standard TypeORM raw query pattern. All acceptable.

## Phase 2 - Security

### 2.1 PASS - Teacher-isolation invariant

All class CRUD and enrollment operations are scoped by currentTeacherId. findOne loads by id and compares cls.teacherId !== currentTeacherId, throwing 404 (never 403) on mismatch. findAll queries with where: { teacherId: currentTeacherId }. enroll() validates both class and student ownership before any write. The e2e suite covers list, single, patch, delete, enrollments GET, and cross-teacher enrollment attempt -- all return 404 not 403.

### 2.2 PASS - Soft deletes

@DeleteDateColumn deletedAt present on both Class and ClassEnrollment. @Exclude() decorates deletedAt on both entities; ClassSerializerInterceptor (global) omits it from responses. The e2e suite asserts the field is absent from serialized output. No hard deletes. No isActive booleans.

### 2.3 PASS - DTO validation, no Zod on transport

All DTOs use class-validator decorators. CreateClassDto validates all fields. EnrollDto uses @IsUUID(4). UpdateClassDto extends PartialType(CreateClassDto). No Zod transport schemas introduced.

### 2.4 PASS - Migration safety

Migration 5-CreateClassesAndEnrollments.ts names every constraint: pk_classes, chk_classes_duration, fk_classes_teacher_id, fk_classes_created_by, fk_classes_updated_by, pk_class_enrollments, fk_class_enrollments_class, fk_class_enrollments_user, uq_class_enrollments_active. AppModule confirms synchronize: false. No edits to previously-merged migrations.

### 2.5 PASS - No try/catch on Postgres error codes

No try/catch in ClassesService or StudentsService. ConflictException thrown from application logic, not DB error code inspection. QueryFailedErrorFilter maps all five new M5 constraints.

### 2.6 PASS - No SQL injection vectors

No raw queries in ClassesService. Both raw queries in StudentsService.validateExclusionFilters use parameterized positional placeholders; user-supplied values are pre-validated as @IsUUID(4) at the DTO layer.

### 2.7 PASS - RBAC at controller class level

@Roles(UserRoleType.TEACHER) at class level in ClassesController (line 42). No per-method overrides. The e2e RBAC test confirms a STUDENT-role JWT receives 403.

### 2.8 PASS - No JWT secret fallback introduced

No new JWT strategies added. Existing JwtStrategy uses configService.getOrThrow -- unchanged.

### 2.9 PASS - No response field leakage

Both deletedAt fields carry @Exclude(). e2e suite asserts these are absent from HTTP responses for classes and enrollments.

### 2.10 WARN - Dependency CVEs: tar@6.2.1 via bcrypt chain

pnpm audit reports 6 high-severity vulnerabilities in tar@6.2.1 (GHSA-34x7-hfp2-rc4v, GHSA-8qq5-rm4j-mr97, GHSA-83g3-92jg-28cx, GHSA-qffp-2rhf-9h96, GHSA-9ppj-qmqm-q256, GHSA-r6q2-hw4h-h46w) via bcrypt@5.1.1 -> @mapbox/node-pre-gyp@1.0.11 -> tar@6.2.1.

**Runtime exposure**: NONE. tar is used only by node-pre-gyp during bcrypt native compilation at install time. Not in the production bundle.

**Exploitation prerequisite**: supply-chain compromise of CI/CD. Build-toolchain risk only; not a web API attack surface.

**Recommendation**: suppress as a known build-time transitive CVE. Track bcrypt releases for a version shipping a patched node-pre-gyp.

## Suggested suppressions

1. **tar@6.2.1 HIGH x6** (Finding 2.10) - build-time transitive CVE via bcrypt -> @mapbox/node-pre-gyp; zero runtime exposure. Accept until bcrypt ships a patched node-pre-gyp.

2. **validateExclusionFilters bare existence check** (Finding 1.4) - acknowledged deferral in code comments per the milestone plan. Accept for now; close in M9 isolation hardening.

3. **as User type cast** (Finding 1.7) - standard TypeORM partial-entity pattern; no safer alternative. Accept.
